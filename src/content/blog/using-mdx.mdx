这套博客已经启用了 `@astrojs/mdx`，所以你可以在文章中混合 Markdown 和组件。但我并不建议把 MDX 当成默认写作格式。

## 什么时候只用 Markdown

如果一篇文章只是：

- 文字说明
- 图片
- 引用
- 代码片段
- 表格和列表

那 Markdown 已经足够，而且更轻、更稳、更方便迁移。

## 什么时候应该用 MDX

当你遇到这些需求时，MDX 就开始体现价值了：

1. 你想复用统一的文章信息块，比如提示卡片、结论块、术语定义块。
2. 你需要嵌入定制 UI 组件，而不是只靠 Markdown 语法。
3. 你未来打算把某些内容块接成 React 交互组件。

换句话说，MDX 适合解决“内容里需要组件”的问题，而不是为了让每篇文章都显得更高级。

## 如果以后需要复杂交互

你已经要求后续复杂交互优先用 React 而不是 Vue，这个判断是合理的。对 Astro 博客来说，更稳的方式通常是：

```mdx
先写静态内容
  -> 找到确实需要交互的模块
  -> 仅在这些局部引入 React 组件
```

这样既保留内容站的加载优势，也不会把所有页面都变成客户端优先。

## 结论

MDX 最好的使用方式不是“全面替换 Markdown”，而是作为一个增强层：在普通文章结构稳定之后，用它承接那些 Markdown 难以优雅表达的部分。
