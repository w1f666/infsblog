很多 React 页面在最初写起来都很顺，直到状态开始从局部按钮、筛选条件、弹窗开关一路扩散到整页，代码才突然变得难以解释。

判断状态边界时，我一般会先问三个问题：这个状态是谁创建的，它会影响哪些区域，以及它是否需要跨层级复用。回答清楚这三个问题，状态大概率就知道该放在哪里了。

如果一个状态只影响当前组件的显示逻辑，那它通常应该留在本地。比如一个折叠面板是否展开、一个输入框是否聚焦，这些都没有必要提到页面级。

但如果一个状态会驱动多个兄弟组件，或者会影响 URL、筛选列表、分页、请求参数，那么它就不再只是局部交互，而是页面状态。这个时候把它抬高，通常能换来更清晰的数据流。

最糟糕的情况不是状态多，而是状态位置和影响范围不匹配。状态一旦被放错地方，后续每次新增需求都像在绕路。

所以真正值得反复练习的不是某一种状态库，而是对边界的判断能力。
