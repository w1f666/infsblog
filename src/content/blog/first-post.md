很多 starter 的问题不在于跑不起来，而在于它只能证明“这个技术栈可以工作”，却不能直接承接真实内容。Astro 默认博客模板就是典型例子：结构合理，但还缺少长期维护需要的组织能力。

这轮整理里，我优先处理了三个层面的事情。

## 1. 先让站点元信息进入生产态

如果网站标题还是默认值，页脚还是模板的社媒链接，关于页还是占位说明，那么不管页面多完整，用户都会立刻意识到这只是一个没写完的样板站。

所以第一步不是堆效果，而是把这些最基础的信息改成真实站点的表达：

- 站点标题、描述、语言和关键词
- 头部导航与 RSS 入口
- 页脚说明与版权信息
- 文章标签、精选文章等内容元数据

这些字段不会直接决定视觉效果，但会决定站点是否像一个真的博客。

## 2. 把列表页做成真正可读的文章目录

博客站点的核心页面通常不是首页，而是文章列表和文章详情。只要这两个页面的节奏、层级和密度不对，整个阅读体验就会显得很散。

我最后选择把文章列表改成纯文本主导的纵向结构，而不是继续强调图片卡片。原因很简单：对技术博客来说，用户真正需要快速看到的是标题、摘要、日期和标签，而不是封面图。

当列表能承担“快速扫描”的角色后，站点的可用性会明显提升。

## 3. 先把内容模型设计好

一个长期写作的博客，迟早会遇到内容增多后的组织问题。所以我在内容模型里优先加了标签和精选字段，让首页、归档页、标签页都能从同一套 frontmatter 里拿数据。

```ts
{
	title: '文章标题',
	description: '简短摘要',
	pubDate: '2026-04-10',
	tags: ['Astro', '写作'],
	featured: true,
}
```

这类字段很轻，但它们会明显降低后续扩展的成本。无论是做标签筛选、专题聚合还是搜索入口，都可以在同一套数据结构上往下走。

## 结论

对于技术博客来说，第一阶段最重要的不是功能数量，而是结构是否能撑住后续持续写作。页面能否清楚表达内容，通常比是否有更多交互更重要。

等内容模型、列表结构和站点信息都稳定之后，再往上叠搜索、筛选或 React 交互模块，会更稳。
