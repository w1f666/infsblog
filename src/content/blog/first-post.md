很多 starter 的问题不在于不能运行，而在于只能演示，不能直接上线。Astro 官方博客模板就很典型：结构是对的，但默认内容、头尾信息、视觉气质和落地细节都还是样板状态。

这次整理我优先解决了三件事。

## 1. 先把站点元信息做真实

如果网站标题还是 `Astro Blog`，页脚里还是官方社媒，关于页还是占位文本，那么页面再好看也会立刻暴露出“模板味”。

所以第一步不是写动画，也不是换框架，而是把这些最基础的信息换成真实站点的表达：

- 站点标题、描述、语言和关键词
- 头部导航与 RSS 入口
- 页脚说明与版权信息
- 文章标签、精选文章等内容元数据

这些信息同时影响用户判断和搜索引擎理解，越早改越值。

## 2. 把“能看”升级到“耐看”

默认模板的样式通常很克制，但也容易太平。我的处理思路是保留内容优先的前提，再增加一点氛围和层次：

- 用更稳定的配色做出轻暖色背景，不让整站像默认白板
- 用卡片式表面组织首页和列表页，让信息分组更明确
- 用更适合标题的衬线字体栈，强化文章站的阅读感
- 把文章页的头图、标签、更新日期和正文宽度统一起来

重点不是“做得花”，而是让页面在没有复杂交互的情况下也足够完整。

## 3. 内容结构必须先跑通

真实博客不是首页截图，而是一组长期会被不断追加的内容系统。为了让后续写作不费劲，我给文章集合扩展了标签和精选字段，并用它们驱动首页与列表页的展示逻辑。

```ts
{
	title: '文章标题',
	description: '简短摘要',
	pubDate: '2026-04-10',
	tags: ['Astro', '写作'],
	featured: true,
}
```

这类元数据很轻，但能明显提升站点组织能力。等文章变多后，你再接入搜索、筛选、专题推荐页，也不需要推翻当前内容结构。

## 一个很实际的结论

个人博客的第一阶段不需要复杂的客户端框架。你更需要的是：

1. 稳定的页面骨架
2. 清晰的内容模型
3. 可直接上线的视觉完成度
4. 后续能低成本维护的目录与组件结构

等这些都稳定后，再局部加 React 去做真正需要交互的模块，才是更稳的节奏。
